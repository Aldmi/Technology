1. Init commit в ветке master
2. Создали Ветку prod
3. Переключились (checkout) обратно на ветку master
4. Сделали рабочие коммиты (Commit1, Commit2)
5. Пришо время делать релиз:
   5.1 Переключились на ветку prod
   5.2 Сделали Merge master->prod  - в ветке prod появится коммит результат операции merge
   ОБЯЗАТЕЛЬНО ОТМЕТИТЬ ГАЛОЧКУ "create a new commit even if fast-forward is possible sourcetree" - ЭТО СОЗДАСТ НОВЫЙ Merge COMMIT
6. Переключились (checkout) обратно на ветку master
7. Работаем в master, пока не пришло время пункта [5]

ИТОГО: в ветке prod будут только комиты версий для production (результаты команды Merge)



Если галочка "create a new commit even if fast-forward is possible sourcetree" не стоит - это режим Ускоренное слияние, в котором если нет конфликтов, то изменения просто сливаются в основную ветку.
Флаг --no-ff
Ускоренное слияние имеет смысл для кратковременных ветвей, но в более сложной истории слияние без ускорения может облегчить понимание истории и упростить возврат группы коммитов.
--no-ff Флаг вызывает слияние всегда создавать новый объект коммита, даже если слияние может быть выполнено с помощью быстрой перемотки вперед.
Это позволяет избежать потери информации об историческом существовании ветви компонента и объединяет все коммиты, которые вместе добавляли функцию.
